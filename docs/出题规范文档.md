# 出题规范文档

> 最后更新：2026-06-25
> 目的：规范 AI 预制题出题流程，保证题量持续增长、质量稳定可控

---

## 一、出题策略总览

### 核心原则

```
只使用预制题和真题
   ↓
预制题由 AI 基于 handbook 数据逐题编写 → 自动校验 → 入库
真题由人工录入真实考卷 → 打标签 → 入库
   ↓
用户练习时从预制题库和真题库中取题，不做运行时实时生成
   ↓
题量不足时通过扩充数据文件解决，而非降级到实时生成
```

### 为什么放弃实时生成？

| | 实时生成（已放弃） | 预制题 + 真题 |
|--|---------|--------|
| 解析质量 | 模板化，深度不够 | 逐题编写，详细透彻 |
| 干扰项 | 随机抽取，合理性不稳定 | 精心设计，区分度高 |
| 好题沉淀 | 随生随弃 | 固定入库，持续积累 |
| 题型覆盖 | 只有 7 种有生成器 | 所有题型均可覆盖 |
| 数据可追溯 | 无 | 标注 handbookSource |
| 运行时风险 | 需加载 63KB 生成器代码 | 纯数据读取，零风险 |

## 二、题库数据源说明

### 2.1 两个数据文件

| 文件 | 内容 | 来源筛选 |
|------|------|---------|
| `exam-questions.json` | 中考真题/期末真题 | `qsrc === 'exam'` |
| `auto-questions.json` | AI 编写的预制题 | `qsrc === 'auto'` |

> ⚠️ `question-bank.js` 中的 GENERATORS（实时生成器）已于 2026-06-24 弃用。
> 答题时只从以上两个 JSON 文件读取数据，不做运行时实时生成。

### 2.2 数据流

```
用户选择来源 → generateUnifiedQuestions()
    ├── "真题"    → _sampleFromExam()  → exam-questions.json
    ├── "预制题"  → _sampleFromAuto()  → auto-questions.json（不足时返回已有题，不降级）
    └── "全部"    → exam + auto 混合
```

### 2.3 auto-questions.json 格式

```json
{
  "version": 1,
  "generated_date": "2026-06-25",
  "total": 200,
  "notes": "预制题库说明",
  "questions": [
    {
      "id": "auto_pinyin_001",
      "type": "pinyin",
      "format": "choice",
      "difficulty": "medium",
      "qsrc": "auto",
      "semester": "all",
      "status": "approved",
      "question": "题干文本（\\n 表示换行）",
      "options": [
        { "text": "选项A", "correct": true },
        { "text": "选项B", "correct": false },
        { "text": "选项C", "correct": false },
        { "text": "选项D", "correct": false }
      ],
      "answer": "正确答案",
      "explanation": "详细解析，包括为什么选这个、为什么其他不对",
      "knowledgePath": "基础知识 > 语音",
      "handbookSource": "语音 > 必考知识梳理 > 初中生容易读错的成语"
    }
  ]
}
```

## 三、出题流程

### 3.1 标准流程

```
1. 选择题型
   ↓
2. 从 handbook 对应章节提取素材
   ↓
3. 编写题目（逐题，非程序化）
   ↓
4. 填写：题干 + 选项 + 答案 + 解析 + handbookSource
   ↓
5. 运行自动校验脚本
   ↓
6. 通过 → 写入 auto-questions.json
   不通过 → 修改后重新校验
   ↓
7. 更新缓存版本号 → 部署
```

### 3.2 出题来源对照

| 题型 | handbook 数据源 | 当前状态 | 建议每批产量 |
|------|---------------|---------|-------------|
| 字音 (pinyin) | 语音 > 必考知识梳理 > 容易读错的成语(397条) / 容易读错的词语(874条) | ✅ 30 题预制 | 20-30 |
| 字形 (char) | 汉字 > 必考知识梳理 > 易混字辨析(119组) / 形近字(200组) | ✅ 20 题预制 | 20-30 |
| 词语运用 (word_usage) | 词语 > 必考知识梳理 > 成语释义 | ✅ 20 题预制 | 20-30 |

## 四、出题规范

### 4.1 通用规则

1. **题干清晰**：让考生明确知道要做什么
   - 选择题："下列...正确/有误的一项是"
   - 判断题："判断...是否正确"
2. **选项设计**：
   - 4 个选项（选择题）/ 2 个选项（判断题）
   - 干扰项要合理，不能一眼看出是错的
   - 选项之间长度不要差距太大
3. **答案准确**：必须与 handbook 数据一致
4. **解析详细**：
   - 先说答案，再说为什么
   - 对每个干扰项简要说明错误原因
   - 对容易混淆的知识点做对比

### 4.2 各题型规范

#### 字音
- 标注方法：在易错字后加括号注音，如 `按捺(nài)不住`
- 干扰项设计：针对常见的误读情况

#### 字形
- 题干要明确是考"错别字"
- 干扰项使用常见的形近字/音近字混淆

#### 词语运用
- 考成语在语境中的使用是否正确
- 避免生僻成语
- 语境要贴近初中生生活

### 4.3 质量检查清单

每道题写完自查：

- [ ] 答案在选项中（选择题）
- [ ] 选项无重复
- [ ] 题干不含占位符（`______`、`undefined` 等）
- [ ] 解析不含"程序生成"等模板残留词
- [ ] 选项长度合理
- [ ] 解析解释了正确答案和干扰项
- [ ] `handbookSource` 标注了数据来源

## 五、自动校验

### 5.1 校验脚本

每次提交前运行：

```bash
node scripts/validate-auto-questions.js
```

### 5.2 校验项

| 检查项 | 说明 |
|--------|------|
| 题干非空 | 不能为空或含 undefined |
| 答案非空 | 不能为空 |
| 选项完整性 | 选择题 ≥ 2 个选项，判断题 = 2 个选项 |
| 正确选项存在 | 至少有一个选项的 correct 为 true |
| 选项无重复 | 所有选项文本不重复 |
| 解析非空 | 长度 ≥ 5 |
| id 唯一 | 所有 id 不重复 |
| status 合法 | 必须为 approved 或 pending |
| qsrc 合法 | 必须为 auto |

### 5.3 生成脚本

复用 `scripts/generate-auto-questions.js` 作为生成+校验一体化脚本。
新题型加入时，在脚本中新增对应的问题数组，然后在底部组装。

## 六、AI 对抗性复核

自动校验只能检查结构问题（空值、重复、格式），无法判断内容对错。
因此入库后需要对题目内容进行对抗性复核。

> **复核策略：逐题复核，不抽样。**
> 预制题没有时效压力，时间充裕，不需要用抽样来推断整体——每一道题都值得单独检查。

### 6.1 复核方式：逐题对抗

用独立 Agent，以"挑错者"视角**逐题**检查，不设样本量、不设阈值。
每批题目入库后，全量逐题复核。

### 6.2 复核维度：四维对抗

| 维度 | 检查内容 | 典型问题 |
|------|---------|---------|
| **答案正确性** | 标注的正确答案是否确实正确？ | 把对的标成错的、多音字判断有争议、单选题有多个正确选项 |
| **干扰项合理性** | 错误选项是否确实错误？ | 模棱两可、也能算对的选项 |
| **解析准确性** | 解析内容本身对不对？ | 解释错误、张冠李戴、解析与答案自相矛盾 |
| **数据源一致性** | 题目与 `handbookSource` 是否吻合？ | 标注来源是"成语"实际考的是"词语" |

每道题判三档：

| 判定 | 含义 |
|------|------|
| ✅ 通过 | 四个维度均无问题 |
| ⚠️ 有瑕疵 | 有小问题但不影响答题（如解析不够详细） |
| ❌ 需修正 | 答案/选项/解析有事实性错误 |

### 6.3 执行流程

```
新题入库 → 逐题四维复核 → 发现 ❌/⚠️ → 修正 → 重新校验 → 入库
```

- ❌ 和 ⚠️ 都需要修正，不区分严重程度——预制题有的是时间打磨
- 修正后重新运行校验脚本，确保没有引入新的结构问题

### 6.4 修正流程

1. 在 `auto-questions.json` 中找到对应 `id` 的题目
2. 修改错误字段
3. 如果问题严重需暂时下线，将 `status` 改为 `pending`，修正后改回 `approved`
4. 修正完成后推送更新

## 七、版本与缓存

### 7.1 版本号规则

每次修改 auto-questions.json 后，更新以下文件中的缓存版本号：

| 文件 | 修改位置 |
|------|---------|
| `js/practice.js:77` | `question-bank.js?v=YYYYMMDDx` |
| `js/practice.js:80` | `question-registry.js?v=YYYYMMDDx` |

版本号格式：`v=YYYYMMDDa` 日期+字母序号，同一天修改递增字母。

### 7.2 题量规划参考

| 阶段 | 描述 |
|------|------|
| 当前 | 12 种题型全覆盖，预制题 + 真题双轨运行 |
| 短期目标 | 扩充各题型题量，每种题型达到充足覆盖 |
| 中期目标 | 真题积累 ≥ 10 套试卷，预制题持续跟随 handbook 新增内容 |
| 理想目标 | 真题 + 预制题总量达到可支持完整模拟卷的水平 |

## 八、FAQ

### Q: 预制题怎么防止重复？

A: 生成脚本内部用 `id` 做唯一标识，出题时检查 `data/auto-questions.json` 已有题目避免重复。

### Q: 预制题出错了怎么办？

A: 找到对应 `id`，在 `auto-questions.json` 中修改后重新提交。如果有用户反馈，可在答题界面加反馈按钮标记。

### Q: 预制题不够怎么办？

A: 通过 `scripts/generate-auto-questions.js` 扩充预制题数量。不应降级到实时生成。

### Q: 怎么加新题型？

A: 1. 在 `question-types.json` 中注册题型 → 2. 在 `scripts/generate-auto-questions.js` 中添加对应的出题数组 → 3. 运行脚本并校验 → 4. 如果该题型在 handbook 中无对应数据源，先录入真题到 `exam-questions.json`
